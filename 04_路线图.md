# 路线图

## 1. 目标场景

白天你负责写论文，晚上系统自动帮你整理格式。  
第二天你拿到的是：

- 已修正的 `docx`
- 修改报告
- 待人工确认问题清单

## 2. v1 阶段能做什么

`v1` 先固定方法，不急着追求一键全自动。  
它的主要目标是让系统具备稳定顺序和可复用规则：

- 先读规范
- 先建样式库
- 先改结构
- 再改正文
- 最后处理图表、文献、引用
- 交付前强制终检

## 3. 理想的自动化流水线

### 输入

- 最新版论文 `docx`
- 格式要求
- 模板或示例稿
- 老师意见

### 自动化任务

1. 扫描今天是否有新增内容
2. 重新读取规范与老师意见
3. 更新规则表
4. 在副本上重建样式库和结构
5. 批量套样式
6. 修复段落与分页
7. 检查图表、表格、题注
8. 检查文献与正文引用
9. 运行终检清单
10. 产出交付包

### 输出

- `最终稿.docx`
- `修改报告.md`
- `待人工确认问题清单.md`
- `日志/版本记录`

## 4. 需要继续建设的模块

### 规则解析模块

把不同学校的格式要求沉淀成可执行规则库。

### 文档重构模块

专门负责：

- 样式库
- 标题层级
- 分节与页码
- 目录导航

### 图表文献模块

专门负责：

- 图题表题
- 三线表
- 截图风险
- 参考文献
- 正文引用

### 终检报告模块

专门负责：

- 最终核验
- 问题清单
- 修改报告

## 5. 自动化的边界

以下内容不应默认全自动决定：

- 是否删除整章
- 是否改动核心论证
- 是否补真实数据
- 是否替换关键文献
- 是否调整案例口径

系统应将其标成"待人工确认"，而不是偷偷修改。

## 6. 最后的方向

未来这套系统成熟后，目标不是"会调字体"，而是：

**把任何一篇学校规范明确的 `docx` 论文，稳定地从混乱状态整理成可提交状态。**
